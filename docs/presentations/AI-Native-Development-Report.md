## Слайд 6: Строгое разделение ролей

### Почему это критично

Каждый агент имеет **жёсткие ограничения**:

| Роль | МОЖЕТ | НЕ МОЖЕТ |
|------|-------|----------|
| **Art Director** | CSS, ассеты, UI | Редактировать .ts файлы |
| **Developer** | Писать код, тесты | Создавать задачи |
| **Reviewer** | Одобрять/отклонять | Исправлять код сам |
| **Architect** | Создавать задачи | Писать код |

### Зачем такие ограничения?

1. **Предотвращение конфликтов** — агенты не мешают друг другу
2. **Чёткая ответственность** — всегда понятно, кто виноват
3. **Предсказуемость** — поведение системы стабильно

## Слайд 7: Zero Trust методология

### Оператор НЕ читает код

Это не лень. Это принцип.

**Традиционный подход:**
```
Developer → Code Review (человек) → Merge
```

**AI-Native подход:**
```
Developer → Reviewer (Opus) → Reviewer (Gemini) → Автотесты → Merge
```

### Как это работает

1. Developer создаёт PR
2. Reviewer (Claude Opus) проверяет логику и архитектуру
3. Reviewer (Gemini) проверяет визуал и UX
4. CI/CD запускает тесты
5. **Всё зелёное** → Оператор нажимает Merge

### Бинарный контроль

Оператор видит только два слова:
- ✅ **APPROVED**
- ❌ **REJECTED**

Никаких нюансов. Никакого "ну, в целом норм, но вот тут бы поправить".

## Слайд 8: Beads — AI-Native Task Management

### Почему не JIRA?

JIRA создана для людей. Веб-интерфейс, кнопки, дропдауны.

Агентам нужен **CLI**.

### Beads — система управления задачами для агентов

```bash
# Посмотреть доступные задачи
bd ready

# Взять задачу в работу
bd update 42 --status=in_progress

# Посмотреть детали
bd show 42

# Закрыть задачу
bd close 42 --reason="Реализовано в PR #123"
```

### Преимущества

| Аспект | JIRA | Beads |
|--------|------|-------|
| **Интерфейс** | Web UI | CLI |
| **Интеграция с Git** | Плагин | Встроено |
| **Зависимости задач** | Сложно | `depends_on: [41, 40]` |
| **Для агентов** | Неудобно | Идеально |

## Слайд 9: Memory Bank — внешняя память агентов

### Проблема

Агенты не помнят предыдущие сессии. Каждый раз начинают с чистого листа.

### Решение

Markdown-файлы как "долговременная память":

```
.memory_bank/
├── activeContext.md      → Что происходит СЕЙЧАС
├── progress.md           → Что ЗАВЕРШЕНО
├── systemPatterns.md     → КАК построена архитектура
├── techContext.md        → КАКИЕ технологии
├── productContext.md     → ЧТО мы строим
├── codebaseIndex.md      → ГДЕ что лежит
├── decisionLog.md        → ПОЧЕМУ так решили
└── projectRoadmap.md     → КУДА движемся
```

### Как это работает

1. В начале сессии агент читает Memory Bank
2. Получает полный контекст проекта
3. В конце сессии **обязательно** обновляет Memory Bank

## Слайд 10: Landing the Plane Protocol

### Критичный протокол завершения сессии

Каждая сессия **должна** закончиться по протоколу:

```
1. ✅ Создать issues для оставшейся работы
2. ✅ Запустить quality gates (npm test)
3. ✅ Обновить статусы задач в Beads
4. ✅ git push + проверка синхронизации
5. ✅ Обновить Memory Bank
```

### Почему "Landing the Plane"?

Пилот не может просто выйти из кабины на высоте 10,000 метров.

Агент не может просто закрыть сессию с незакоммиченным кодом.

### Гарантия

**Работа НИКОГДА не потеряется между сессиями.**

## Слайд 12: Технические достижения — Детерминизм

### Проблема мультиплеера

В реалтайм-игре важно, чтобы все клиенты видели одинаковую картинку.

### Решение: 100% детерминированная симуляция

| Параметр | Значение |
|----------|----------|
| **Тиков в секунду** | 30 |
| **Детерминизм** | 100% |
| **Строк тестов** | 3,948 |

### Что запрещено в коде симуляции

```typescript
// ❌ ЗАПРЕЩЕНО
Math.random()
Date.now()
performance.now()

// ✅ РАЗРЕШЕНО
import { Rng } from './utils/rng';
const rng = new Rng(seed);
rng.next();  // Детерминированное случайное число
```

### Как это обеспечивается

Автотесты на каждом PR проверяют:
- Нет ли `Math.random()` в коде симуляции
- Воспроизводится ли игра по seed

## Слайд 16: Проблемы — God Objects

### Что пошло не так

```typescript
// ArenaRoom.ts — 3,749 строк, 126+ методов
class ArenaRoom {
  handleInput() { ... }
  processMovement() { ... }
  checkCollisions() { ... }
  applyDamage() { ... }
  spawnOrbs() { ... }
  // ... ещё 120 методов
}
```

### Почему это произошло

1. Developer не видит "большую картину"
2. Каждая задача добавляет методы в существующий класс
3. Без человека-архитектора код растёт хаотично

### Решение

Декомпозиция на 15 систем (в процессе):

```
ArenaRoom
├── MovementSystem
├── CollisionSystem
├── DamageSystem
├── SpawnSystem
└── ... (11 систем)
```

## Слайд 17: Проблемы — Scope Creep

### Типичная ситуация

**Задача:** "Добавить кнопку выхода из игры"

**Developer делает:**
1. Добавляет кнопку выхода ✅
2. Рефакторит UI-систему ❌
3. Добавляет анимацию кнопки ❌
4. Оптимизирует рендеринг ❌

### Почему это проблема

- Задача разрастается в 4 раза
- Review занимает в 4 раза больше времени
- Вероятность багов растёт

### Решение

Правило **"одна задача = один фокус"**:

```markdown
# В описании задачи
[SCOPE]
- Только кнопка выхода
- Без рефакторинга
- Без оптимизаций
- Без "улучшений по пути"
```

## Слайд 18: Проблемы — Merge Conflicts

### Когда это происходит

Два агента работают параллельно над разными задачами, но трогают один файл.

```
Agent A: изменяет строки 100-150 в ArenaRoom.ts
Agent B: изменяет строки 120-180 в ArenaRoom.ts
                    │
                    ▼
              CONFLICT!
```

### Решение

1. **PR workflow** — каждая задача = отдельная ветка
2. **Человек мержит** — разрешает конфликты вручную
3. **Мелкие задачи** — меньше изменений = меньше конфликтов

### Статистика

За 30 дней было **12 серьёзных конфликтов**. Все разрешены оператором.

### Детерминированный RNG

```typescript
// server/src/utils/rng.ts
export class Rng {
  private state: number;

  constructor(seed: number) {
    this.state = seed;
  }

  next(): number {
    // Mulberry32 алгоритм
    let t = (this.state += 0x6d2b79f5);
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  }

  nextInt(min: number, max: number): number {
    return Math.floor(this.next() * (max - min + 1)) + min;
  }
}
```
