**Источник истины для Slime Arena (не менять):**
1) `SlimeArena-Architecture-v1.4.md` — границы модулей, сетевые принципы, серверная симуляция, техстек.
2) `SlimeArena-Plan-v1.7.md` — этапы работ и критерии готовности.
3) `SlimeArena-GDD-v2.3.md` — правила игры и требования к интерфейсу.

## 0) Правила переноса
1) Не переносить доменную часть u2 (космос/полет/специфичные режимы).
2) Переносить только инфраструктурные паттерны: симуляция 30 Hz, модель ввода, сглаживание, наблюдаемость, тестирование, дисциплина документов, процессы релиза.
3) Любой перенос, влияющий на архитектуру или сетевой контракт, оформлять ADR в `docs/ADR/`.
4) При переносе не противоречить `SlimeArena-Architecture-v1.4.md`. Если есть сомнение — фиксировать вопрос в ADR как «открытый».
5) Результат переноса должен быть проверяемым: критерии готовности + минимальный набор тестов (unit/integration/e2e) согласно разделам тестирования в архитектуре и плане.

## 2) Поиск по ключам
Искать в `u2/docs` по словам и именам сущностей:
- `30 Hz`, `fixed timestep`, `dt`, `tick`
- `PlayerInput`, `one per tick`, `input`
- `snapshot`, `state`, `interpolation`, `extrapolation`, `smoothing`, `blend`
- `authoritative`, `validation`, `rate limit`, `anti-cheat`
- `torus`, `wrap`, `unwrapped`, `continuous coordinates` (u2) + `bounds`, `walls`, `arena` (то, что нужно для Slime Arena)`
- `network simulator`, `latency`, `loss`, `jitter`
- `metrics`, `observability`, `logging`
- `tests`, `integration`, `e2e`, `regression`
- `CI`, `workflows`, `docker`, `registry`, `release`

Дополнительно (полезно для Slime Arena):
- `config`, `tuning`, `balancing` (конфигурируемые параметры симуляции)
- `compat`, `backward`, `versioning` (совместимость контрактов)

## 4) Шаблон извлечения (на каждый найденный документ)
Для каждого прочитанного файла u2 создать заметку по форме:
- **Источник (u2):** путь файла.
- **Тезис:** 1–3 предложения (что важно).
- **Почему полезно Slime Arena:** 1–2 предложения.
- **Как применить в Slime Arena (без реализации):** какие модули затронет (`client/`, `server/`, `shared/`, `docs/`, `tests/`) и какой ожидаемый эффект.
- **Критерии готовности:** 2–5 проверяемых пунктов.
- **Какие тесты нужны:** unit/integration/e2e.
- **Риски:** производительность, сеть, безопасность, сложность поддержки.

## 5) Обязательные паттерны (итоговые результаты)
ИИ‑агент должен подтвердить, что в u2 docs найдено и понято (или явно зафиксировано, что не найдено):
1) **Fixed timestep 30 Hz:** почему выбрано и какие ограничения.
2) **Модель ввода «one per tick» (`PlayerInput`):** почему лучше пакетирования.
3) **Server authoritative:** что строго считается на сервере и как валидируется ввод.
4) **Client smoothing:** интерполяция, ограниченная экстраполяция, сглаживание коррекций.
5) **Topology & bounds:** если u2 описывает torus/wrap — зафиксировать как *не применимо*; для Slime Arena подтвердить правила **бортов** (отражение/коррекция, отсутствие «проскальзывания» и «залипания»), и какие тесты это покрывают.
6) **Network simulator:** профили плохой сети и повторяемость дефектов.
7) **Test strategy:** минимум unit + integration + e2e, дисциплина регрессий.
8) **Docs discipline:** как у них связаны изменения кода и обновления документов.
9) **Release/CI/CD:** практики сборки, публикации, отката.
10) **Наблюдаемость:** какие метрики и логи обязательны.

## 6) Как оформлять перенос в Slime Arena
### 6.1 ADR
Если паттерн влияет на архитектуру/протокол — создать ADR в `docs/ADR/` и сослаться на соответствующий фрагмент u2 (путь файла).

### 6.2 Задачи
Создать задачи в трекере проекта:
- 1 паттерн = отдельный набор задач (крупная задача + подзадачи).
- В каждой задаче указать критерии готовности и минимальные тесты.

### 6.3 PR‑требования
- В описании PR указать источники (пути файлов u2).
- Описать, как проверить результат (шаги проверки без кода).
- Приложить короткое видео (или запись экрана) с демонстрацией эффекта.
- Указать, какие тесты добавлены/обновлены.

---

## 7) Критерии успешной «экстракции u2 docs»
Считается выполненным, если:
- есть список минимум 10 тезисов из u2 docs (с путями файлов);
- по каждому тезису есть решение: перенести/не переносить + почему;
- по переносимым — заведены задачи и/или ADR;
- минимум один паттерн внедрён и подтверждён проверками (тесты + демонстрация).

---

## 8) Запрещённые результаты
- Копирование больших фрагментов текста без адаптации и без ссылок на источник.
- Изменение сетевого контракта без ADR и без стратегии совместимости.
- Внедрение паттерна без критериев готовности и без тестов (кроме мелких правок интерфейса, если это явно обосновано).
