## 1. Топология 1: Docker Compose на одном хосте

### 1.1. Описание

Все компоненты размещены на одной VPS в Docker-контейнерах. `MetaServer` — один экземпляр. `MatchServer` — N экземпляров на разных портах. `Redis` и `PostgreSQL` — общие.

### 1.2. Структура сервисов

`REQ-SCALE-060`: Предоставить файл `docker-compose.multi.yml` (или расширить существующий `docker-compose.yml`) со следующими сервисами:

| Сервис | Количество | Порт(ы) | Описание |
|--------|-----------|---------|----------|
| `postgres` | 1 | 5432 | База данных |
| `redis` | 1 | 6379 | Кэш, очередь, реестр |
| `meta` | 1 | 3000 | MetaServer |
| `match-1` | 1 | 2567 | MatchServer, экземпляр 1 |
| `match-2` | 1 | 2568 | MatchServer, экземпляр 2 |
| `match-N` | 1 | 2567+N-1 | MatchServer, экземпляр N |
| `client` | 1 | 5173 | Статика клиента |
| `admin` | 1 | 5175 | Админ-панель |

`REQ-SCALE-061`: Каждый сервис `match-*` получает уникальные переменные:

| Переменная | match-1 | match-2 | match-N |
|------------|---------|---------|---------|
| `MATCH_SERVER_ID` | `ms-1` | `ms-2` | `ms-N` |
| `PORT` | `2567` | `2568` | `2567+N-1` |
| `MATCH_SERVER_PUBLIC_HOST` | публичный IP/домен | публичный IP/домен | публичный IP/домен |
| `MATCH_SERVER_PUBLIC_PORT` | `2567` | `2568` | `2567+N-1` |

Остальные переменные (`META_SERVER_URL`, `MATCH_SERVER_TOKEN`, `REDIS_URL`, `JWT_SECRET`) — одинаковые для всех экземпляров.

### 1.3. Ограничения

- Вертикальное масштабирование: ограничено ресурсами одной машины.
- Все `MatchServer` делят CPU и память хоста.
- При выходе из строя хоста — все серверы недоступны.

### 1.4. Когда подходит

- Начальное масштабирование (Stage 1-2).
- Нагрузка до 1000-2000 CCU (2-4 MatchServer на одном хосте).
- Бюджет ограничен одним VPS.

## 2. Топология 2: Отдельные VPS

### 2.1. Описание

`MetaServer`, `Redis` и `PostgreSQL` размещены на основном хосте (или каждый на своём). `MatchServer` — на отдельных VPS. Каждый `MatchServer` подключается к общим `Redis` и `MetaServer` по сети.

### 2.2. Структура

| Хост | Сервисы | Описание |
|------|---------|----------|
| Основной (VPS-0) | MetaServer, Redis, PostgreSQL, Client, Admin | Центральная инфраструктура |
| Game-1 (VPS-1) | MatchServer-1 | Игровой сервер |
| Game-2 (VPS-2) | MatchServer-2 | Игровой сервер |
| Game-N (VPS-N) | MatchServer-N | Игровой сервер |

### 2.3. Конфигурация MatchServer на отдельном VPS

| Переменная | Значение | Описание |
|------------|----------|----------|
| `MATCH_SERVER_ID` | `ms-{номер}` | Уникальный идентификатор |
| `MATCH_SERVER_PUBLIC_HOST` | Публичный IP VPS | Адрес, видимый клиентам |
| `MATCH_SERVER_PUBLIC_PORT` | `2567` | Порт (стандартный, один на VPS) |
| `REDIS_URL` | `redis://{IP основного хоста}:6379` | Подключение к Redis по сети |
| `META_SERVER_URL` | `http://{IP основного хоста}:3000` | Подключение к MetaServer |
| `MATCH_SERVER_TOKEN` | Общий секрет | Авторизация server-to-server |
| `JWT_SECRET` | Общий секрет | Верификация joinToken |

### 2.4. Сетевые требования

`REQ-SCALE-070`: Redis-порт (6379) доступен только для `MatchServer` VPS. Использовать firewall (iptables, VPC security groups) для ограничения доступа.

`REQ-SCALE-071`: Порт `MatchServer` (2567) открыт для публичного доступа (клиенты подключаются напрямую). Рекомендуется TLS-терминация (см. §4).

### 2.5. Когда подходит

- Нагрузка свыше 2000 CCU или ограничения по CPU одного хоста.
- Требование отказоустойчивости: выход одного game-хоста не влияет на другие.
- Географическое распределение серверов (разные датацентры).

## 3. Health checks и мониторинг

### 3.1. Heartbeat-мониторинг на MetaServer

`REQ-SCALE-080`: `MetaServer` при обработке matchmaking-запросов проверяет наличие ключа `matchserver:{serverId}:alive` в Redis. Серверы с истёкшим TTL исключаются из пула.

Активная проверка (периодический poll) не требуется для P0 — достаточно реактивной проверки при каждом запросе matchmaking.

### 3.2. Алерты

`REQ-SCALE-081`: Если число активных серверов в реестре стало меньше ожидаемого (например, `EXPECTED_MATCH_SERVERS`), `MetaServer` логирует предупреждение уровня `WARN`:

```
[MatchServerRegistry] WARN: Active servers: 1/3 (expected 3)
```

Интеграция с внешними системами алертинга (Telegram-бот, email) — вне scope P0.

### 3.3. Метрики для TZ-MON

Расширение мониторинг-панели (`TZ-MON-v1.6`) — отдельная задача. Данное ТЗ определяет API, которые панель сможет использовать:

- `GET /api/v1/admin/servers` — список серверов с метриками
- `GET /api/v1/admin/stats` — агрегированные показатели
- `GET /api/v1/admin/rooms` — комнаты со всех серверов

## 4. TLS для прямых подключений

### 4.1. Проблема

Клиент в production работает через HTTPS. Браузер требует `wss://` (WebSocket Secure) для подключения к `MatchServer`. При прямом подключении по `roomHost:roomPort` каждый `MatchServer` должен поддерживать TLS.

### 4.2. Варианты решения

| Вариант | Сложность | Описание |
|---------|-----------|----------|
| A. Nginx TLS-терминатор на каждом хосте | Средняя | Nginx перед `MatchServer`, TLS-сертификат на Nginx. Клиент подключается к `wss://{host}:443` (или другой порт) |
| B. Wildcard-сертификат с поддоменами | Средняя | `ms1.slime-arena.overmobile.space`, `ms2.slime-arena.overmobile.space`. DNS A-записи указывают на IP каждого VPS. Let's Encrypt wildcard через DNS-01 challenge |
| C. Единый Nginx reverse proxy | Низкая | Все `MatchServer` за одним Nginx. Роутинг по path-префиксу (`/ws/ms1/`, `/ws/ms2/`). Клиент использует единый домен. Ограничение: bottleneck на прокси при высокой нагрузке |

Рекомендация: вариант B для production с отдельными VPS, вариант C для Docker Compose на одном хосте.

## 5. Graceful drain и rolling update

### 5.1. Процедура drain

1. Оператор вызывает `POST /api/v1/admin/servers/{serverId}/drain` через админ-панель.
2. `MetaServer` проксирует вызов на `POST /api/internal/drain` целевого `MatchServer`.
3. `MatchServer` обновляет статус в реестре на `draining`.
4. `MatchmakingService` перестаёт направлять новые матчи на этот сервер.
5. Существующие комнаты продолжают работу. Каждый матч длится 90 сек (match) + 15 сек (results + restart). Максимальное время ожидания: ~2 минуты.
6. Когда `roomCount = 0`, оператор может безопасно остановить контейнер.

### 5.2. Rolling update (обновление без простоя)

1. Drain сервер A (§5.1).
2. Дождаться `roomCount = 0` на сервере A.
3. Остановить контейнер A, обновить образ, запустить.
4. Сервер A регистрируется в реестре, получает новые матчи.
5. Повторить для сервера B, C, ... N.

### 5.3. Аварийное отключение

При аварийном отключении `MatchServer`:

1. Ключ `matchserver:{serverId}:alive` исчезает по TTL (30 секунд по умолчанию).
2. Новые матчи перестают направляться на этот сервер.
3. Игроки в активных комнатах теряют текущий матч. Клиент показывает «Соединение потеряно» и возвращает в лобби после таймаута reconnect (15 сек).
4. `MatchSummary` для незавершённых матчей не отправляется. Награды за эти матчи не начисляются.

## 6. Сценарии деплоя

### 6.1. Масштабирование с 1 до 2 MatchServer

**Предусловие:** Stage 1 реализован (реестр, self-registration, клиентский роутинг).

1. Добавить сервис `match-2` в Docker Compose (или поднять VPS-2).
2. Задать уникальные `MATCH_SERVER_ID` и `MATCH_SERVER_PUBLIC_PORT`.
3. Запустить контейнер.
4. Убедиться: сервер появился в `GET /api/v1/admin/servers`.
5. Убедиться: новые матчи распределяются между серверами.

### 6.2. Вывод сервера из эксплуатации

1. Drain (§5.1).
2. Дождаться `roomCount = 0`.
3. Остановить контейнер.
4. Удалить сервис из Docker Compose (если навсегда).

### 6.3. Горизонтальное масштабирование (N серверов)

Повторить §6.1 для каждого нового сервера. При значительном количестве серверов (10+) рекомендуется автоматизировать через скрипт или CI/CD pipeline.
