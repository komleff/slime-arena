# SDD-Lite: Фреймворк документации для Slime Arena

**Версия:** 1.0 | **Статус:** Draft | **Дата:** 2026-02-17

------

## 1. Зачем это нужно

Slime Arena разрабатывается с активным участием AI-агентов (кодеры, дизайнеры, ревьюеры). Каждый агент работает в ограниченном контексте без памяти между сессиями. Это создаёт три повторяющихся проблемы:

- **Дрейф терминов** — разные агенты называют одни и те же сущности по-разному, порождая несогласованный код.
- **Потеря решений** — причины архитектурных выборов живут в головах или в переписке и теряются через неделю.
- **Скрытые пробелы** — неописанные элементы (как ClassSelectModal в UI-TZ v1.1) обнаруживаются случайно при ревью или, хуже, при реализации.

SDD-Lite — минимальный набор правил и шаблонов без бюрократии. Только Markdown в репозитории и дисциплина обновления.

## 4. Шаблоны

### 4.1. Блок метаданных

Добавить в начало каждого документа:

```yaml
---
title: Название документа
version: X.Y
status: draft | review | approved | deprecated
created: YYYY-MM-DD
updated: YYYY-MM-DD
author: человек / AI-модель
depends_on:
  - docs/specs/GDD-Core.md
depended_by:
  - docs/specs/UI-TZ.md
---
```

Правила заполнения:

- `draft` — в работе, может содержать пробелы
- `review` — передан на проверку (человеку или другой модели)
- `approved` — проверен, пригоден для агентов-кодеров
- `deprecated` — устарел, указать чем заменён
- `depends_on` — если эти документы изменятся, данный нужно проверить
- `depended_by` — если данный изменится, эти нужно проверить

### 4.2. Глоссарий (glossary.md)

```yaml
---
title: Глоссарий Slime Arena
version: 1.0
status: draft
updated: 2026-02-17
---
```

#### Правила использования

1. Каждый термин имеет одно каноническое название.
2. В коде используется значение из колонки «Идентификатор».
3. При создании нового документа или кода — сверяться с глоссарием.
4. Новый термин добавляется сюда ДО использования в спецификациях.

| Термин     | Идентификатор | Определение                                                  | Источник          |

| Охотник    | `Hunter`      | Класс слайма. Пассивка: +15% к скорости. Стартовое умение: Рывок | GDD-Abilities п.2 |

### 4.4. Запись решения — ADR (decisions/_TEMPLATE.md)

```yaml
---
title: "ADR-NNN: Краткое название решения"
status: proposed | accepted | deprecated | superseded
date: YYYY-MM-DD
superseded_by: ссылка, если заменено
---
```

**Структура:**

**Контекст** — что произошло, какая проблема или развилка. Какие ограничения и требования влияют на выбор.

**Варианты** — вариант A, B, C. Каждый с плюсами и минусами.

**Решение** — какой вариант выбран и почему.

**Последствия** — что меняется в проекте. Какие документы и модули затронуты.

**Связанные документы** — ссылки на specs, gaps, другие ADR.

#### Пример заполненного ADR

```yaml
---
title: "ADR-001: Telegram Mini App как платформа"
status: accepted
date: 2026-01-05
---
```

**Контекст:** Нужно выбрать платформу распространения для MVP. Требования: минимальный CPI, без установки, быстрый запуск, мультиплеер.

**Варианты:**

- Вариант A — нативное приложение (iOS + Android): полный доступ к устройству, но стоимость ×2, ревью сторов, высокий CPI.
- Вариант B — PWA: один код, без стора, но ограниченные push-уведомления, нет виральности.
- Вариант C — Telegram Mini App: виральность через чаты, без установки, встроенные платежи, аудитория Telegram. Минусы: ограничения WebView.

**Решение:** Вариант C. На этапе MVP важнее охват и виральность, чем производительность. Ограничения WebView приемлемы для 2D-игры.

**Последствия:** UI учитывает безопасные зоны Telegram WebView. Запрет нативной прокрутки (overflow hidden). Платежи через Telegram Stars / TON. Тестирование только внутри Telegram.

**Связанные документы:** docs/specs/UI-TZ.md (ограничения WebView), docs/gaps.md — GAP-002 (потеря соединения).

### 4.6. Шаблон спецификации (specs/_TEMPLATE.md)

```yaml
---
title: Название спецификации
version: 1.0
status: draft | review | approved | deprecated
created: YYYY-MM-DD
updated: YYYY-MM-DD
author: человек / AI-модель
depends_on:
  - список документов
depended_by:
  - список документов
---
```

**1. Назначение** — одно-два предложения: что описывает и зачем.

**2. Область действия** — что входит. Что явно НЕ входит.

**3. Связь с глоссарием** — все термины должны быть в glossary.md. Новые — сначала в глоссарий, потом сюда.

**4. Основное содержание** — специфично для каждого документа.

**5. Открытые вопросы** — перенести в gaps.md:

| Вопрос | Почему важен | Предложенные варианты |
| ------ | ------------ | --------------------- |
| ...    | ...          | ...                   |

**6. История изменений:**

| Версия | Дата       | Автор | Что изменилось |
| ------ | ---------- | ----- | -------------- |
| 1.0    | YYYY-MM-DD | ...   | Первая версия  |

## 5. Инструкция по внедрению

### Шаг 1: Создать структуру (30 минут)

1. Создать `docs/` в корне репозитория со структурой из раздела 3.
2. Скопировать шаблоны в соответствующие папки.
3. Создать `glossary.md` — заполнить терминами из GDD-Core и GDD-Abilities.
4. Создать пустой `gaps.md`.

### Шаг 2: Миграция существующих документов (1–2 часа)

1. Перенести GDD-Core, GDD-Abilities, UI-TZ, ScreenMap в `docs/specs/`.
2. Добавить блок метаданных в начало каждого.
3. Заполнить `depends_on` и `depended_by` — пройтись по ссылкам внутри документов.
4. Проверить, что все термины есть в глоссарии. Отсутствующие — добавить.

### Шаг 3: Зафиксировать известные решения (1 час)

Создать ADR для решений, которые уже приняты:

- ADR-001: Telegram Mini App как платформа
- ADR-002: Частота обновления HUD (5–10 Гц)
- ADR-003: Запрет прокрутки в WebView

Для каждого: контекст, что решили, почему. Кратко.

### Шаг 4: Собрать пробелы (30 минут)

Пройтись по всем документам и вынести в `gaps.md`:

- всё, помеченное как TODO, TBD, «уточнить»
- отсутствующие элементы (как ClassSelectModal)
- допущения без проверки (баланс классов)

## 6. Рабочие сценарии

### 6.1. Создание нового документа (новая фича / модуль)

```
1. Проверить glossary.md — есть ли нужные термины.
   Нет → добавить в глоссарий.

2. Создать файл в docs/specs/ по шаблону.
   Заполнить метаданные, depends_on.

3. Написать содержание. Нерешённые вопросы →
   секция «Открытые вопросы».

4. Перенести вопросы в gaps.md с ID.

5. Обновить depended_by в документах-зависимостях.

6. Ревью (другой AI-моделью или самому через день).

7. status → approved. Передать агенту-кодеру.
```

### 6.2. Передача задачи AI-агенту

Что давать агенту:

- **Обязательно:** конкретная спецификация + glossary.md
- **По необходимости:** связанные ADR, спецификации из depends_on, записи из gaps.md (чтобы агент знал, чего НЕ надо придумывать)

В промпте: «Используй только термины и идентификаторы из глоссария. Встретил неописанную сущность — остановись и сообщи.»

### 6.3. Изменения во время кодирования

Ситуация: при реализации обнаружено, что спецификация неполна или противоречива.

```
1. СТОП. Не додумывать, не импровизировать.

2. Зафиксировать:
   — мелкое уточнение → комментарий // SPEC-QUESTION: описание
   — влияет на логику → запись в gaps.md

3. Можно продолжить без ответа → продолжить, пометив место.

4. Нельзя (развилка, два варианта) → остановить задачу,
   описать варианты, передать человеку.

5. После решения:
   — обновить спецификацию (version++, дата, история)
   — закрыть запись в gaps.md
   — проверить depended_by — нужно ли обновить зависимые
```

### 6.4. Изменения после плейтеста

```
1. Записать наблюдения по шаблону changelogs/.

2. Классифицировать каждое наблюдение:
   — баг → в трекер задач (вне docs/)
   — проблема баланса → обновить GDD-Abilities, добавить GAP
   — проблема UX → обновить UI-TZ или создать новую спецификацию
   — архитектурная развилка → создать ADR

3. Каскад обновлений. Для каждого изменённого документа:
   — version++, updated, история изменений
   — проверить depended_by
   — если зависимый тоже затронут → обновить

4. Обновить glossary.md (новые термины, изменённый смысл).

5. Обновить gaps.md:
   — закрыть подтверждённые допущения
   — добавить новые вопросы
   — перевести допущения в факты или проблемы

6. Git-коммит: «docs: playtest YYYY-MM-DD — описание»
```

### 6.5. Ежемесячная проверка согласованности

```
[ ] Все термины в specs/ и коде совпадают с glossary.md
[ ] Нет записей в gaps.md старше 30 дней без движения
[ ] Статусы документов актуальны
[ ] depends_on / depended_by отражают реальные связи
[ ] Нет ADR со статусом proposed старше 14 дней
[ ] Все SPEC-QUESTION в коде имеют парную запись в gaps.md
```

Проверку можно поручить AI-агенту: скормить все docs/ и попросить найти рассогласования.

## 7. Правила для AI-агентов

Добавить в системный промпт или файл правил:

```
ПРАВИЛА РАБОТЫ С ДОКУМЕНТАЦИЕЙ:

1. Перед началом работы прочитай glossary.md.
   Используй ТОЛЬКО термины и идентификаторы из глоссария.

2. Если встречается сущность, которой нет в глоссарии —
   ОСТАНОВИСЬ и сообщи.

3. Не принимай архитектурных решений самостоятельно.
   Спецификация не покрывает случай — зафиксируй и спроси.

4. Противоречие между документами — следуй более свежему.
   Зафиксируй противоречие как GAP.

5. Спорные места в коде помечай:
   // SPEC-QUESTION: описание вопроса
```

## 8. Чего НЕ делать

- **Не строить полную матрицу трассируемости.** depends_on / depended_by достаточно. RTM оправдана для команд 10+.
- **Не заводить ADR на мелочи.** Только решения, которые: а) можно было принять иначе, б) влияют на несколько модулей, в) забудутся через месяц.
- **Не требовать PR-ревью документации.** Git — да, формальный процесс — нет. Достаточно кросс-проверки другой AI-моделью.
- **Не дублировать содержание.** Ссылка вместо копии. Исключение — глоссарий (краткое определение допустимо).
- **Не обновлять ради обновления.** Только при реальном изменении: фича, плейтест, пробел, решение.
